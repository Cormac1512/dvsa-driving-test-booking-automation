## 2025-02-21 - Bot Stealth & Default Data Exfiltration
**Vulnerability:** Submitting default placeholder data ("15/08/2024", "PS2 4PZ") to the target site due to missing validation checks in the `enterTestDate` and `enterPostcode` functions.
**Learning:** Even if data matches a valid format (e.g. a valid date regex or postcode regex), it must also be validated against default/placeholder values to prevent the script from executing with generic junk data. Submitting identical placeholder data across many sessions increases the likelihood of bot detection and IP bans.
**Prevention:** Implement strict equality checks against default placeholder constants (e.g. `DEFAULT_DATE`, `DEFAULT_POSTCODE`) in addition to format validation regexes before submitting form data.
## 2026-03-04 - [Preventing Unhandled Exceptions in setTimeout Callbacks]
**Vulnerability:** The randomDelay function executes DOM interaction callbacks inside a setTimeout without any try-catch block. If an error occurs (e.g., trying to access a property on a null element if the DOM changed between scheduling and execution), the exception goes unhandled, potentially leaking the stack trace to the console, revealing the automation script's presence.
**Learning:** Any scheduled execution (setTimeout, setInterval, requestAnimationFrame) needs explicit error handling within the callback wrapper, as errors thrown asynchronously do not bubble up to the caller's try-catch blocks. In an automation context, unhandled errors can trigger anti-bot telemetry.
**Prevention:** Wrap asynchronously executed callbacks in a try-catch block and handle errors gracefully (e.g., logging safely without leaking the full stack trace).
## 2026-03-05 - [Preventing Unhandled Exceptions in setInterval Callbacks]
**Vulnerability:** Similar to `setTimeout`, callbacks executed within `setInterval` bypassing the caller's execution context and try-catch blocks. Specifically, the `countdown` utility in the DVSA script lacked explicit error handling for the `onTick` and `onComplete` callbacks. If a DOM element vanishes or an external error occurs during the countdown, an unhandled exception could leak the stack trace to the browser console.
**Learning:** In browser automation, *all* asynchronous timers (both single execution like `setTimeout` and recurring like `setInterval`) must have their internal callback execution wrapped in explicit try-catch blocks to prevent leaking presence. Recurring timers also need to ensure the interval is cleared upon error to prevent endless error loops.
**Prevention:** Wrap `setInterval` internal callback executions in try-catch blocks, catch exceptions gracefully (without leaking stack traces), and explicitly call `clearInterval` upon catching an error to stop further execution.
## 2026-04-10 - Weak RNG Fallback for Large Delays
**Vulnerability:** The `randomIntBetween` function fell back to `Math.random()` when the requested range was >= 2^32. This fallback used a weak, predictable random number generator, enabling potential bot detection logic to fingerprint the script and predict exact interaction delays.
**Learning:** Rejection sampling using standard `Uint32Array` values is limited by the 32-bit maximum. Simply reverting to `Math.random()` for larger delays negates the secure environment entirely. Instead, a larger limit (like `Number.MAX_SAFE_INTEGER`, roughly 2^53) can be generated mathematically using consecutive draws of a 21-bit high word and a 32-bit low word.
**Prevention:** Construct 53-bit ranges securely by leveraging two standard 32-bit buffer draws per loop, shifting and masking appropriately. Apply rejection sampling against the true 53-bit range threshold to preserve entropy.
## 2026-05-15 - [Preventing Unhandled Exceptions in Nested Async Callbacks and requestAnimationFrame]
**Vulnerability:** The `showToast` function used `requestAnimationFrame` and deeply nested `setTimeout` callbacks without explicit error handling. If DOM elements disappeared or were modified externally, errors in manipulating toast styling or removing the toast from the DOM would go unhandled. This bypasses the primary execution context's `try...catch` blocks, causing stack traces to leak to the console and revealing the script's presence.
**Learning:** In browser automation, simply wrapping a top-level function in a `try...catch` does not protect asynchronous callbacks (like `requestAnimationFrame` or nested `setTimeout`s) triggered within that function. Every distinct asynchronous execution context that interacts with the DOM must have its own internal error handling to prevent stack trace leaks that trigger anti-bot telemetry.
**Prevention:** Wrap the internal logic of *all* asynchronous callbacks (including `requestAnimationFrame` and nested `setTimeout` chains) in explicit `try...catch` blocks to ensure errors are caught and logged securely without exposing stack traces.
## 2026-03-11 - [Preventing 32-bit Integer Overflow in Delay Configurations]
**Vulnerability:** Passing a delay value greater than 2147483647 (the maximum 32-bit signed integer) to JavaScript timer functions (`setTimeout`, `setInterval`) causes an integer overflow, resulting in a nearly instant (1ms) execution. This could lead to a rapid execution loop and act as a Denial of Service (DoS) vulnerability.
**Learning:** Input validation must not only check the lower bound (e.g., `>= 1000`) but also enforce a strict upper bound to ensure the provided parameter is safe for the specific runtime API it will be passed to.
**Prevention:** Implement an explicit maximum upper bound check (`<= 2147483647`) on delay configurations before passing them to timer functions.
